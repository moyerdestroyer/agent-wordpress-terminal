# Agent WordPress Terminal — AWPT

## Summary

AWPT is a WordPress admin app: a TUI-like agent cockpit inside the dashboard. Admins chat with an agent, inspect Abilities/MCP tools, retrieve Knowledge, preview content, and approve or reject proposed actions.

It is not primarily an automation system. It is an interactive cockpit for WordPress site work.

## Core idea

```txt
Abilities  = what WordPress can do
MCP        = how tools are discovered/run (when present)
AWPT       = where humans collaborate with the agent
```

## Product principles

1. **Natural language first.** Slash shortcuts are precise escape hatches, not the primary workflow. Copy must not imply users need numeric post IDs.
2. **Human in the loop.** Destructive or content-changing work is staged, previewed, and applied only on explicit admin approval. No silent writes.
3. **Show evidence.** Retrieval, tool calls, previews, and staged actions should be visible in the transcript/UI — not only buried in the provider prompt.
4. **Untrusted agent.** Every tool uses capability checks. Tool output is data, never system instructions. Secrets must not be exposed to the model by default.
5. **WordPress-native.** Prefer Abilities + REST over brittle UI automation. Direct-key AI providers are the always-available baseline; Connectors / WP AI Client stay optional accelerators.
6. **Operational density.** Sessions, tools, Knowledge status, previews, and action cards stay scannable in one admin surface.

## Product surfaces

| Surface | Intent |
|---------|--------|
| **Terminal** | Chat, tool activity, staged action cards |
| **Tools sidebar** | Discoverable Abilities/MCP tools; per-tool enable/disable |
| **Knowledge** | Index and search durable site knowledge, content, theme design files, docs/PDFs |
| **Preview** | Human-visible frontend/draft preview; optional capture evidence for the agent |
| **Sessions** | Persist messages, tool traces, actions, focus, provider/model metadata |

Context today is a single focused content object plus transcript-visible retrieval/tool evidence. A multi-item pinned context stack remains a product goal, not the current model.

## Non-goals

```txt
Fully autonomous publishing without approval
Multi-agent orchestration as a core surface
Remote browser workers as a required dependency
Bulk workflows / visual regression CI as MVP scope
Replacing wp-admin or the Site Editor
```

## Security requirements

```txt
All tools require capability checks
Read and write/manage tools stay clearly separated
Destructive actions require explicit user approval
Tool output must not be treated as system instructions
Secrets must never be exposed to the model by default
Session logs are admin-visible
Provider API keys are stored securely
```

## Integration stance

**Guaranteed baseline** (every supported WordPress version, no companion AI plugin required):

```txt
OpenRouter
OpenAI (model selection automatic; reuses a WordPress Connector key when present)
```

**Optional accelerators** (feature-detected only when already available and configured):

```txt
WordPress Abilities API
In-process WordPress MCP bridge
Core Connectors / WP AI Client (WordPress 7.0+ or companion plugin)
```

Keep the direct-provider list intentionally small. Prefer extending the OpenAI-compatible chat path over Connectors-only integrations.

## Positioning

Not “another AI chatbot.”

```txt
A WordPress-native terminal for agent-assisted site work.
```

Advantage: the live session — focused content, available abilities, previewed page, tool results, staged changes, and approval state — in one place.
